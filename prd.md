一、核心架构设计

1.1 四层架构体系

输入层：多模态文档解析，支持飞书 PRD、技术文档、Figma 设计、Wiki 页面、历史用例等输入源。自动提取文档结构、识别引用关系、归一化为统一文本格式。
上下文研究层：智能需求理解核心，采用 Lead-Worker 架构。Lead Researcher 负责测试信号打标、研究计划制定；Sub Researcher 执行深度研究，使用互联网搜索、代码库查询、API 文档检索等工具。输出结构化知识图谱和综合上下文文本。
用例生成层：基于子图架构，包含四个有序节点：场景规划员（功能点拆解）、证据映射员（原文关联）、草稿撰写员（用例扩写）、结构装配员（树状组装）。生成结构化测试用例。
反思优化层：质量保障机制，通过内部冗余检查、语义相似度去重、业务规则合规验证、最终计划组装，提升用例质量。
1.2 状态管理机制

全局状态：工作流数据中枢，包含 PRD 输入、中间产物（知识图谱、功能点列表、证据映射）、配置参数（业务规则、模板、语言）、元数据（会话 ID、执行状态、错误日志）。
子图状态：用例生成专用状态，隔离复杂中间产物，确保主工作流清晰。
1.3 工作流节点设计

InputParserNode：文档获取器工厂，按 URL 模式选择对应解析器，提取结构化内容，追踪引用文档。
ContextResearchNode：Lead-Worker 调度器，测试信号分类（行为、状态、条件、异常、UI 反馈、约束、模糊），深度研究循环（提问 - 检索 - 验证 - 沉淀），知识图谱构建（功能主题、用户场景、需求原子、全局约束、模糊点）。
CaseGenNode：子图协调器，封装场景规划、证据映射、草稿撰写、结构装配四个子节点，管理状态转换和数据传递。
ReflectionNode：质量优化器，执行冗余分析、相似度聚类、规则验证、计划组装。
二、核心功能模块

2.1 多模态文档解析

类型识别：基于 URL 模式自动识别飞书文档、Figma、网页、技术文档等
结构化提取：解析飞书文档块结构（标题、段落、表格、代码、评论），转换 Figma 图层为描述文本
引用追踪：自动发现并获取相关引用文档，确保需求完整性
内容归一化：统一输出为 LLM 可消费的文本格式，保留来源元数据
2.2 智能需求理解

测试信号打标：七类信号识别（行为、状态变化、条件、异常、UI 反馈、约束、模糊）
深度研究循环：Lead Researcher 智能调度，Sub Researcher 工具集（搜索、代码查询、API 文档）
知识图谱构建：五类实体提取（功能主题、用户场景、需求原子、全局约束、模糊点），关系建立（包含、实现、约束、澄清）
上下文生成：结构化文本输出，包含 PRD 概览、全局约束、模糊点、功能详情
2.3 测试用例生成

场景规划：基于业务模板的功能点拆解，规则扫描（敏感业务识别），模板对齐，自省修正
证据映射：三轮定位算法（锚定证据、覆盖度检查、深度补强），原文引用验证，置信度评分
用例扩写：三类场景生成（正常、边界、异常），结构化输出（标题、前置条件、操作步骤、预期结果、测试数据、优先级）
结构装配：树状数据组装，符合目标系统格式要求
2.4 质量优化保障

内部冗余检查：步骤语义相似度分析，重复操作合并，验证点集中
语义去重：基于嵌入向量的用例聚类，相似度阈值控制，智能合并策略
规则合规验证：业务规则知识库匹配，违规用例标记 / 修正，优先级处理
综合质量评估：覆盖度、完整性、效率、维护成本、风险覆盖多维度指标
2.5 业务适配支持

模板管理系统：三层模板结构（基础、业务线、项目），声明式描述语言，智能匹配算法
多语言支持：语言识别管道，三种适配模式（完全翻译、关键词保留、混合），术语库管理
本地化场景：文化差异考虑，区域特性适配，本地化测试数据生成
配置管理：分层配置体系（基础、业务、环境），动态加载机制，安全保护
三、技术实现要点

3.1 LangGraph 工作流设计

节点编排：顺序执行与条件分支结合，支持异步并发处理
状态驱动：全局状态共享，子图状态隔离，数据一致性保证
错误处理：重试机制、降级策略、监控报警、自动恢复
性能优化：缓存策略、连接复用、批量处理、资源调度
3.2 智能代理架构

Lead Researcher：整体需求理解、测试蓝图构建、任务调度、质量评估
Sub Researcher：专项问题研究、工具使用、证据收集、信息验证
协同机制：任务分解、结果整合、冲突解决、质量反馈
3.3 数据处理管道

文档预处理：格式转换、内容提取、结构分析、元数据标记
信息抽取：实体识别、关系建立、证据定位、置信度计算
用例生成：模板填充、数据合成、结构验证、格式输出
质量优化：冗余检测、相似度计算、规则检查、报告生成
四、开发实施内容

4.1 核心模块开发

状态管理模块：GlobalState 和 CaseGenState 模型定义，状态持久化接口，数据验证逻辑
文档解析模块：飞书文档解析器、Figma 转换器、网页提取器、引用追踪器
研究代理模块：Lead Researcher 调度逻辑、Sub Researcher 工具集成、知识图谱构建器
用例生成模块：场景规划算法、证据映射引擎、用例扩写模板、结构装配器
质量优化模块：冗余检查算法、相似度聚类器、规则验证器、评估指标计算
4.2 工作流实现

主工作流：四节点顺序编排，状态传递机制，错误处理管道
子图工作流：用例生成四阶段流水线，状态隔离设计，结果回传逻辑
异步处理：并发任务调度，资源池管理，结果聚合机制
监控体系：执行追踪、性能指标、质量评估、报警通知
4.3 业务集成

模板配置：业务模板定义语言，动态加载机制，版本管理
规则引擎：业务规则解析器，合规性检查逻辑，违规处理策略
多语言支持：语言识别管道，翻译服务集成，术语库管理
输出适配：目标系统格式转换，数据验证，批量导出
五、关键优化特性

5.1 从文档中识别的优化需求

Case 内部冗余优化：通过优化 Prd Parsing 层、引入 Case Reflection 机制、优化生成结构，减少无关测试内容。效果：用例从 362 个减少到 95 个1。
Case 间重复优化：通过动态生成 Case 层级、Case 语义相似度去重，减少重复测试点。效果：用例从 95 个优化到 82 个，另一文档从 78 个优化到 57 个1。
文档内容权重打分：训练文档内容权重模型，提取重要内容，减少无关信息影响。效果：用例从 82 个优化到 57 个，另一文档从 104 个优化到 70 个1。
业务模板支持：与业务一对一沟通，定制化业务模板，满足不同业务编写习惯1。
5.2 架构设计优势

模块化可扩展：四层架构清晰，组件职责明确，支持功能扩展
智能需求理解：Lead-Worker 架构实现深度研究，确保需求理解准确性
质量闭环保障：从输入到输出的全流程质量检查，提升用例实用性
业务适配灵活：模板化配置支持不同业务需求，降低定制成本
技术栈现代化：基于 LangGraph 的工作流设计，支持异步并发、状态管理、错误恢复
